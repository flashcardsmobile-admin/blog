Why Lisp?

In most of my personal projects, I use the Common Lisp programming language. This is often seen as an odd decision since Lisp has been around longer than my parents have been alive. Surely it can't have anything to bring to the table, right? Lisp in fact has three main advantages: a mature ecosystem, an arsenal of compilers, and unrivalled metaprogramming.

A Mature Ecosystem

Lisp may not have many developers, but it has had a half-century for its few users to refine and perfect many useful libraries, some of which have no parallels in other languages. To see the highlights of the modern Lisp ecosystem, one only has to look at the Awesome Common Lisp list. Within, you will see a variety of the normal libraries that one would expect an ecosystem to have, web servers, CSV/YAML/TOML/&c parsers, Unit Testing frameworks, and so on.

Some may wonder why there are so many modern libraries when Lisp is a niche language. I hypothesize that since Lisp is so niche, many who discover it and use it regularly are dedicated enough developers to do lots of open-source work. It may also be that the small community naturally improves the quality of pull requests; there are only so many people who know this library exists and those who do are sufficiently competent to intentionally learn a 50 year-old programming language.

It must also be noted that Lisp has many libraries that few other languages have. Due to the flexibility of the language, which will be detailed later, it has extensions which add logic programming, JSX-style HTML generation, a web scripting language, pattern matching, and a fast regex compiler.

It must thus be concluded that the Common Lisp language has a mature ecosystem usable for a variety of projects. As an added benefit, Common Lisp does not have an "official" package manager; while QuickLisp is the defacto standard, it is also just as easy to use something like Vend to vendor your dependencies if you are concerned about package manager supply chain attacks.

An Arsenal of Compilers

Common Lisp does not have an official implementation. It has an ANSI standard that ensures different compilers and interpreters can all run a certain subset of common functionality. Where the standard has gaps, there have been defacto standards created by packages that bridge the differences between implementations. Due to this, there are a variety of notable implementations with excellent use-cases.

The fastest implementation is the compiler-only Steel Bank Common Lisp (SBCL). It is very performant and makes heavy use of type annotations when optimizing code. It compiles directly to machine code and has an incremental garbage collector. Because of this speed, it is a good choice for servers.

Another notable implementation is the interpreter/compiler Embeddable Common Lisp (ECL). It is notable since its compiler translates Lisp code into C code. Because of this, it is able to target any machine C can, practically everything. Due to the transliteration, it is less performant than SBCL.

Another notable implementation is Armed Bear Common Lisp (ABCL). It compiles its code into bytecode for the Java Virtual Machine, which allows it to interoperate with Java programs and Java libraries effortlessly. If you work in an enterprise shop where everything has to be connected to Java, this can be pitched as a "fast scripting language" for rapidly prototyping concepts or sketching out app internals.

We can now conclude that Common Lisp has quite a few compilers available to ensure code runs on whatever platform in whatever manner you wish it to. notably, there are seldom many tweaks and adjustments needed to make code work between implementations unless you explicitly use implementation-specific features such as ABCL's Java Interop or SBCL's atomics. There are many more compilers and interpreters for this language, but this assortment should give some scope to the reader.

Unrivalled Metaprogramming

All the above features are nice, but those are things any language could hypothetically have. The core feature that sets all lisp languages apart is metaprogramming. This point will require some code examples to prove, so please bear with me as I try to explain enough of Lisp that the argument is cogent.

First, in Lisp, a function/operator call is in the form of (operator operand operand operand) such as (+ 1 1) ; 2 or (* 3 3 3) ; 27 (the semicolons are comments). This notation works whether you're adding numbers, invoking loops, or making functions. The whole syntax is uniform that way. For example, to make a linked list, you just write (list 1 2 3 4). If you're making more of a tree-like structure, you can just write: (list 1 2 (list 3 4 5) 6 "lists aren't all the same type").

Because Lisp programmers use a lot of linked lists when prototyping, there is a shorthand for writing lists, the single-quote: '(1 2 3 4 5). For a tree: '(1 2 3 (4 5) (6 7)). Notice we didn't need more single-quotes when we were writing the interior lists. The single quote essentially switches on "data mode" for everything within the parentheses. Now, compare these two: '(+ 1 1) and (+ 1 1). Lisp code is all linked-lists.

This similarity is inside the compiler too. At compile-time, the AST of Lisp code internally is represented by the same kind of linked lists that our code can use. Due to this, there are many things that can be done via Common Lisp's macro system.

Conclusion

In conclusion, Lisp is worth using in the modern day because of its mature ecosystem, arsenal of compilers, and unrivalled metaprogramming. It can morph into whatever language best fits the problem if you take the time to master it and learn to shape it. I currently use it for web development, such as for my senior project . You may notice that the design is not that great. That's why I'm using Scrimba to teach myself UI design with CSS for my next project.

More Resources

To get a good overview of Lisp's other strengths, see Conrad Barksi's humorous promotional site for his introductory Lisp book. The music video is top-notch ( Direct YT Link).

Barski is also creator of the "Lisp Alien" unofficial logo.

Once you've completed Barski's work, I would recommend reading Paul Graham's Lisp Book, On Lisp. He also has many essays and musings about Lisp on his site, so check those out. Notably, he made his own lisp dialect to create Hacker News.

While Graham's work is good, he does differ from normal CL style in his books. If you want to see a more standard CL code style and some more application, see Peter Norvig's Paradigms of Artificial Intelligence Programming.
